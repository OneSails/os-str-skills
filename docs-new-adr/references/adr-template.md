## 1. ADR 触发条件（何时写）

**满足任一即写**：
- 更换核心技术栈或数据库（如 Go → Rust、Postgres → MySQL）
- 改变系统边界或事实来源（如引入新的事实来源、改变服务边界）
- 改变安全或鉴权策略（如引入 OAuth、改密钥管理方式）
- 做出明确的技术债妥协（如「暂不实现 X，因为 Y」，并接受其代价）

**通常不写**（劝退，除非用户坚持）：
- 纯实现细节（用什么设计模式、函数怎么拆）
- 可逆的常规选型（用哪个 lint 工具、命名风格）
- 无长期影响的小改动

> 判断尺度：半年后有人问「当初为什么这么定？」，若答得上且答案重要 → 写 ADR；若答案是「随便选的 / 显而易见」→ 不写。

## 2. ADR 文件模板

```markdown
---
adr: 0001
status: accepted        # proposed | accepted | deprecated | superseded
date: 2026-06-13
superseded-by: ""
---

# ADR-0001: {简短标题}

## Context
为什么需要这个决策。

背景、驱动因素、面临的约束、要解决的问题。

## Decision
决定怎么做。

具体方案，足够明确到可执行。

## Consequences
收益、代价、风险、后续约束。

- 收益：……
- 代价：……
- 风险：……
- 后续约束：这个决策对未来有什么限制

## Alternatives Considered
考虑过但未采用的方案。

- 方案 A：……，未采用因为……
- 方案 B：……，未采用因为……
```

> `adr` 序号必须与文件名前缀一致（`ADR-0001` ↔ `0001-...-....md`）。
> `date` 用 ISO 格式 `YYYY-MM-DD`。

## 3. 四段写作指引

### Context（背景）
回答「为什么现在要做这个决策」。写：触发事件、当前痛点、不可妥协的约束。**不要写解决方案**——Context 是问题域。

### Decision（决定）
写「我们决定 X」。要具体、可执行、无歧义。读者看完能复述「哦，他们选了 X」。一句话能说清就一句话；复杂决策分点。

### Consequences（后果）
诚实写代价。这条最容易被省略，也最有价值。包含：
- **正面**：得到什么
- **负面**：牺牲什么、新增什么复杂度
- **风险**：什么情况下会反悔
- **后续约束**：这个决定让未来哪些事变难/变易

### Alternatives Considered（备选方案）
写「还考虑过 Y、Z，但没选」。重点是**为什么没选**——这能防止未来有人重提已被否决的方案时，无需重新论证。即使最终只认真考虑了一个备选，也写下来。

## 4. 状态机与取代链

`status` 取值与流转：

```text
proposed ──→ accepted ──→ deprecated
              │
              └──→ superseded（被新 ADR 取代）
```

| status | 含义 | 何时用 |
| --- | --- | --- |
| `proposed` | 提议中，未定稿 | 草稿、待评审 |
| `accepted` | 已采纳，当前有效 | 决策落地 |
| `deprecated` | 已废弃，无直接替代 | 不再适用但无新决策 |
| `superseded` | 被新 ADR 取代 | 必须填 `superseded-by` |

### 取代一条旧 ADR

新建 ADR-0007 取代 ADR-0003 时：

1. 新文件 `0007-...-....md`：
   - frontmatter `status: accepted`、`date: <今天>`
   - Context 开头写：「取代 [ADR-0003](./0003-...-....md)，因为……」
2. 旧文件 `0003-...-....md`：
   - `status: accepted` → `superseded`
   - `superseded-by: "ADR-0007"`
3. 序号不回收：0003 的文件保留，不删除、不改号。

> 序号连续性比「整洁」更重要——追溯链不能断。
